# 复盘：用 AI 做一个 AI API 价格计算器

这次小项目我做的是一个 AI API 价格计算器。它的目标很直接：让用户输入每月请求量、平均输入 token、平均输出 token 和缓存输入占比，然后快速估算 OpenAI、Anthropic、Google Gemini 等海外 AI API 的月度成本。这个工具不复杂，但它解决的是一个真实问题：当我要在多个模型之间做技术选型时，价格往往不是一句“便宜”或“贵”能说清楚的，需要结合自己的请求量和 token 结构来算。

我选择方向 A，是因为它不依赖私有 API key，也不需要后端服务，适合在较短时间内做成一个稳定可访问的静态页面。技术栈我选了原生 HTML、CSS 和 JavaScript，没有使用 React 或 Next.js。这样做的原因很现实：这个项目的核心价值在于数据来源、计算逻辑、移动端体验和部署可访问性，而不是复杂工程结构。少一层构建工具，就少一个可能出问题的地方。

AI 在这个过程中帮了我很多。最开始我让 AI 帮我拆解任务，把招聘题目里的要求整理成清单：公网 URL、GitHub 公开仓库、真实 commit 历史、英文 README、AI 协作记录、复盘博客和 5 行自评。之后我用 AI 生成了页面初版结构，包括输入表单、结果表格、数据来源区域和移动端样式。这个阶段 AI 的价值很明显，它让我不用从空白文件开始一点点搭页面。

但 AI 也给过不可靠的结果。最明显的一次是模型价格数据。AI 会根据记忆给出一些看起来合理但可能过期的价格，甚至会混用不同时间点的模型名称。我发现这个问题，是因为价格数据和官方页面不完全一致。后来我把判断标准改成只信官方价格页面，并且在页面和 README 里写清楚来源链接和查询日期。这个过程让我意识到，AI 可以加快查资料和写代码的速度，但涉及事实和数字时，最终责任还是在自己。

我卡得比较久的地方是移动端表格。价格对比天然有很多列，包括服务商、模型、输入价格、输出价格和月度估算。如果强行把所有列压进手机宽度，文字会非常拥挤，也容易溢出。最后我没有把它改成一堆复杂卡片，而是保留表格结构，并在外层加横向滚动。这样手机上不会撑破页面，用户也能完整查看每一列。

现在我仍然不完全确定的是缓存价格的简化方式。不同厂商对缓存读、缓存写、批量请求、长上下文和不同模型版本的计费规则并不完全一致。这个工具里把缓存输入占比做成一个统一滑块，适合粗略估算，但不适合作为正式财务预算。真正用于生产前，还需要把每个平台的特殊计费规则拆开处理。

如果再给我 8 小时，我会继续做三件事。第一，把价格数据从 JavaScript 代码里抽成单独的 JSON 文件，方便后续维护。第二，增加几种预设场景，比如“低成本客服机器人”“代码助手”“长文档总结”，让用户不用从零输入参数。第三，增加导出功能，把当前估算结果导出成 CSV，方便和同学或团队成员讨论。

这次最大的收获是：AI native 不是把所有判断都交给 AI，而是知道哪些事情可以让 AI 快速生成，哪些事情必须自己验证。页面能跑只是第一步，数据来源是否可信、移动端是否能看、仓库是否有真实提交记录、部署链接是否能打开，这些细节才决定一个小工具是不是真的完成了。
