# 复盘：用 AI 做一个 AI API 价格计算器

这次小项目我做的是一个 AI API Price Calculator，目标很简单：让用户输入每月请求量、平均输入 token、平均输出 token 和缓存命中比例，然后比较 OpenAI、Anthropic、Google Gemini 几个模型的大致月成本。它不是一个复杂产品，但我希望它能解决一个真实的小问题：当我要在几个海外 AI API 之间选型时，先粗算成本，而不是反复翻价格表。

我一开始选择方向 A，是因为它不依赖私有 API key，部署成静态网页就能跑，比较适合在 48 小时内稳定交付。技术栈也刻意选得很轻：HTML、CSS、Vanilla JavaScript。这样做的好处是没有构建链，也没有依赖安装失败的风险，部署到 Vercel / Netlify / Cloudflare Pages 都很直接。

AI 帮我最多的地方是拆任务和做初版结构。我先让 AI 帮我把作业要求拆成可交付清单：公网 URL、GitHub 仓库、英文 README、AI 协作记录、复盘博客和自评。然后我让它生成一个静态页面原型，包括输入控件、结果表格、移动端布局和 README 草稿。这个环节节省了很多从空白文件开始的时间。

AI 也出过错。最明显的一次是价格数据容易过期，模型会凭记忆给出旧价格，甚至混用不同模型版本。我的处理方式是只把官方页面作为数据来源，并在页面和 README 里写清楚查询日期。对我来说，这比“看起来很完整”的数据更重要，因为招聘方会点开链接验证，错了就是错了。

过程中我卡得比较久的是移动端表格。价格对比天然有很多列，如果强行压到手机宽度，文字就会挤在一起。最后我没有把表格做成复杂卡片，而是保留表格语义，并给外层加横向滚动，保证手机上不溢出页面，也能读清楚每一列。

我现在仍然不完全确定的是缓存价格的简化方式。不同平台对缓存读写、批处理、长上下文、模型版本的计费规则不完全一样。这个项目里我把缓存输入作为一个统一比例处理，适合粗估，但不适合直接作为财务预算。如果继续做，我会把每家平台的特殊计费规则拆开，并加上“高级模式”。

如果再给我 8 小时，我会加三件事：第一，支持导出估算结果为 CSV；第二，增加“低成本 / 高质量 / 低延迟”几个预设场景；第三，把价格数据抽成 JSON，并显示每条数据最后验证时间。这样维护起来更清楚，也更接近真实工具。

这次最大的收获是：AI 可以把从 0 到 1 的摩擦降得很低，但关键判断还是要自己做。尤其是数据来源、部署验证、移动端体验这些地方，不能只相信 AI 的回答。一个能跑、能解释、能被验证的小工具，比一个堆满功能但没有上线的项目更有价值。
